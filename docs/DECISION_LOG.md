# Decision Log

Reverse chronological record of non-obvious documentation and product decisions that materially shape this doc system.

| Date | ID | Decision | Why it exists |
| --- | --- | --- | --- |
| 2026-04-18 | `DL-016` | `apps.site` carries `label = "core"` in its `AppConfig`. All migrations for this app use `"core"` as their app label, not `"site"`. Any new migration dependency on this app must reference `("core", "<migration_name>")`, not `("site", ...)`. The label exists because `SiteSettings` and `AboutProfile` originally lived in `apps/core`. When the models moved to `apps/site`, the migration graph was preserved in place rather than squashed. `apps.core` uses `label = "core_runtime"` to avoid the collision. | Renaming the label would require squashing or rewriting every migration in the graph. The alias is intentional and documented here to prevent future contributors from introducing the wrong dependency label. |
| 2026-04-12 | `DL-015` | `make check-content` is a buyer pre-launch gate, not a template release-tag gate. | The template intentionally ships with starter/demo content for evaluation and customization. Requiring the current database to look launch-ready would confuse template release proof with buyer launch proof. The command remains mandatory before any real site goes live. |
| 2026-04-02 | `DL-014` | The repo uses a conventional Django app layout: `apps/core` stays small and cross-cutting, `apps/site` owns site-wide content/config/admin behavior, `apps/pages` owns page composition, and the shared shell stays repo-level. | Root-level app-like packages created too much ceremony. Moving everything app-like under `apps/` restores a normal Django reading experience while still keeping `core` out of junk-drawer territory. |
| 2026-04-02 | `DL-013` | The product stays repo-local at the top level; architecture tightening focuses on clear app ownership rather than turning domain apps into pluggable packages. | The current repo already has clean domain apps, shared shell assets, and strong repo-level QA/docs/ops. Extracting or “pluginising” the product apps now would add churn and blur ownership faster than it would create real reuse value. |
| 2026-03-31 | `DL-012` | Local smoke verification is only considered trustworthy when it targets a prod-like instance rather than a plain `DEBUG=True` dev server. | The smoke script checks the branded 404 surface. Django's technical 404 page in development can fail that check even when the shipped production 404 template is correct, so the release process needs a prod-like target for this step. |
| 2026-03-31 | `DL-011` | Ruff and mypy release gates cover app code, tests, and operational smoke logic, but intentionally exclude ad hoc audit, screenshot, and scratch helper scripts. | Those scripts support manual review and local experiments, but they are not part of the shipped product path. Leaving them in the default static gates made the release signal noisier without improving launch trust. |
| 2026-03-31 | `DL-010` | Release readiness keeps three separate gates: content readiness, deploy/environment readiness, and live smoke verification. | These checks answer different launch questions. Treating them as one generic “release check” makes it too easy to pass environment validation while still missing buyer-facing content issues, or to pass content checks without proving the live routes and assets actually respond. |
| 2026-03-31 | `DL-009` | Public image markup only emits intrinsic `width` and `height` when real metadata is available from the current image objects; the app does not invent placeholder dimensions. | This hardens CLS without lying in the markup or coupling the templates to fake defaults. It also keeps the pass compatible with the current direct-upload image pipeline instead of pretending a responsive-image system already exists. |
| 2026-03-30 | `DL-008` | Project detail hero and share media resolve through one shared `cover_image → first gallery image` fallback path instead of separate cover-only rules. | The project list and related cards already had a richer preview fallback. Reusing one resolved detail-media path prevents the hero and share image from drifting apart and avoids duplicating fallback logic across the detail surface. |
| 2026-03-30 | `DL-007` | The contact flow keeps a single thank-you route, but the page copy branches between `sent` and calm saved-only states while valid enquiries are always saved first. | This preserves the simple public flow and the DB-first safety guarantee without pretending notification delivery succeeded when `CONTACT_EMAIL` is missing or email delivery fails. |
| 2026-03-30 | `DL-006` | `Site Settings` admin mirrors the same readiness logic used by `check_content_readiness` instead of maintaining a second launch checklist. | The buyer-facing admin summary and the pre-launch terminal check need the same source of truth; separate rule sets would drift quickly. |
| 2026-03-30 | `DL-005` | The docs system records `Covered`, `Partial`, and `Gap` states explicitly instead of implying complete coverage. | The repository already mixes automated tests, smoke checks, and visual audit scripts. Hiding that distinction would overstate confidence. |
| 2026-03-30 | `DL-004` | `TECHNICAL_SPEC.md` stays a gateway and delegates detailed rules to the owning spec. | The project already has shared specs and component specs; duplicating them in a master file would create drift quickly. |
| 2026-03-30 | `DL-003` | Component specs use named state IDs that map 1:1 into `qa/TEST_MATRIX.md`. | This keeps required UI states and evidence aligned and makes regressions easier to review in PRs and release checks. |
| 2026-03-30 | `DL-002` | The new doc system describes current implemented behavior, not aspirational future behavior. | The repo already had narrative docs that were ahead of the current homepage implementation. The new system is meant to reduce that drift, not repeat it. |
| 2026-03-30 | `DL-001` | Legacy operational docs remain in place for now and are flagged as external to the locked structure rather than being folded in immediately. | They may still be useful operationally, but forcing them into the new structure during rollout would add scope and mix concerns. |
